# Invoice OCR 发票管理系统

Invoice OCR 是一套可私有化部署的企业发票管理系统，用于集中保存发票原件，通过 OCR 提取结构化数据，并完成项目归档、人工校对、重复检查和数据导出。

系统适合由管理员统一部署，普通用户、财务和管理员通过浏览器协作使用。业务数据、上传原件、OCR 凭据和导出文件均保存在自己的部署环境中。

## 主要能力

- 上传 PNG、JPG、JPEG 或单页 PDF 发票，并异步执行 OCR 识别
- 编辑发票字段与明细行，处理待确认、识别失败和疑似重复记录
- 使用私有项目、共享项目和系统内置的“未分类”项目整理资料
- 将 PDF、图片、DOCX、XLSX 作为普通项目文件归档，无需 OCR
- 按项目、状态等条件筛选发票，并导出 XLSX、CSV 或 JSON
- 将一个项目的发票原件和普通文件打包为 ZIP 下载
- 管理内部用户、固定角色、OCR 运营商配置和额度提醒
- 记录登录、配置和业务操作，便于审计追踪

## 角色与权限

| 角色     | 可见数据             | 项目能力                   | 管理能力                              |
| -------- | -------------------- | -------------------------- | ------------------------------------- |
| 普通用户 | 自己上传的发票和文件 | 创建私有项目，使用共享项目 | 可校对、确认和导出自己的数据          |
| 财务     | 全部用户的发票和文件 | 创建私有或共享项目         | 跨用户校对、确认、重复项处理和导出    |
| 管理员   | 全部用户的发票和文件 | 创建私有或共享项目         | 拥有财务能力，并可管理用户和 OCR 设置 |

系统初始化时创建的第一个账号自动成为管理员。公开初始化入口随后关闭，其他账号只能由管理员在“用户管理”中创建。

## 快速部署

### 1. 准备环境

部署主机需要安装 Docker Engine 和 Docker Compose，并能够拉取以下镜像：

- 应用镜像：`docker.io/nathan974/invoice-ocr-app:0.2.5`
- PostgreSQL：`postgres:16.3-bookworm`
- Redis：`redis:7.2-bookworm`

默认应用镜像面向 `linux/amd64` 以及 `linux/arm64`。在项目目录中执行后续命令。

### 2. 修改生产密钥

打开 `docker-compose.yml`，至少替换以下三个默认值：

| 配置                        | 位置                     | 用途                                  |
| --------------------------- | ------------------------ | ------------------------------------- |
| `POSTGRES_PASSWORD`         | `x-postgres-credentials` | PostgreSQL 数据库密码                 |
| `OCR_CONFIG_ENCRYPTION_KEY` | `x-app-environment`      | 加密数据库中的 OCR 凭据，至少 32 字节 |
| `APP_SECRET_KEY`            | `app.environment`        | 签发登录会话                          |

可以使用 OpenSSL 生成随机值：

```bash
openssl rand -base64 24
openssl rand -base64 32
openssl rand -hex 32
```

三个命令的结果应分别用于数据库密码、OCR 加密密钥和应用密钥。请妥善保存这些值，不要把修改后的生产 Compose 文件提交到公开仓库。

如需修改访问端口，只改 `ports` 左侧的宿主机端口，容器端口保持 `8080`：

```yaml
ports:
  - '18080:8080'
```

### 3. 启动服务

使用默认镜像启动：

```bash
docker compose up -d
```

### 4. 检查服务状态

```bash
curl -fsS http://localhost:8080/healthz
curl -fsS http://localhost:8080/readyz
```

`healthz` 返回 Web 服务状态；`readyz` 同时检查 PostgreSQL 和 Redis。浏览器无法打开系统时，也可以先执行：

```bash
docker compose ps
docker compose logs --tail=200 app worker postgres redis
```

## 首次使用

### 创建首位管理员

新数据库第一次打开时会显示“创建首位管理员”。填写姓名、邮箱和至少 12 位的密码后提交，系统会完成初始化并自动登录。

如果页面直接显示登录表单，说明当前数据库已经初始化。请使用已有账号登录，不要通过删除数据卷重新初始化生产环境。

### 配置 OCR

管理员进入“设置”，创建 OCR 运营商配置并填写 `SecretId`、`SecretKey`、地域、QPS 和额度信息。当前系统暂时只适配腾讯云增值税发票 OCR 配置。

OCR 凭据提交后会在后端加密保存，页面只显示配置状态和凭据指纹，不会再次回显明文。建议先保存并启用配置，再上传一张测试发票确认识别正常。

如果暂时没有 OCR 凭据，上传发票时可以关闭“上传后自动识别”，仅保存原件。当前界面不能为这类原件补建 OCR 作业；后续需要识别时，应重新上传文件并开启自动识别。

### 创建团队账号

管理员进入“用户管理”创建账号，并按职责分配“普通用户”“财务”或“管理员”角色。管理员还可以：

- 修改姓名、部门、角色和启用状态
- 为忘记密码的用户重置密码
- 停用离职或暂时不需要访问的账号

系统要求至少保留一名启用中的管理员。每位用户也可通过右上角用户菜单修改自己的密码，新密码至少 12 位。

## 日常使用指南

### 1. 用项目整理发票和资料

进入“发票库”，在左侧“项目索引”中创建项目：

- **我的私有项目**：仅创建者以及财务、管理员可访问
- **共享项目**：所有已登录用户可使用，仅财务和管理员可创建
- **未分类**：系统内置项目，不可编辑或归档；上传时未指定项目的发票会进入这里

项目可编辑名称和说明，也可归档后再恢复。归档项目不会删除其中的发票或文件。

### 2. 上传发票进行识别

进入“上传识别”，选择“发票 OCR”模式、目标项目和可选的费用场景，然后拖入或选择文件。系统支持逐个上传，也支持一次加入多个文件后批量上传。

发票文件限制：

| 项目     | 限制                                               |
| -------- | -------------------------------------------------- |
| 格式     | PNG、JPG、JPEG、PDF                                |
| PDF 页数 | 仅单页                                             |
| OCR 大小 | Base64 编码后不超过 10 MB，原文件建议小于约 7.5 MB |
| 图片尺寸 | 宽和高均为 20 至 10000 像素                        |
| 不支持   | GIF、扩展名与实际内容不一致的文件                  |

上传队列会显示校验、上传、排队、识别和失败状态。失败项可直接重试；处理较大的队列时，请保持 `worker` 容器正常运行。

### 3. 归档普通项目文件

在上传页切换到“项目文件”模式，可以保存不需要 OCR 的辅助资料。支持 PDF、PNG、JPG、JPEG、DOCX、XLSX，单文件最大 50 MB。

项目文件可在“发票库”切换到“项目文件”视图后预览、下载或删除。删除操作需要再次点击确认。

### 4. 查找和筛选

“发票库”支持按项目、发票号码、发票代码、销售方、购买方、状态和重复标记筛选。顶部全局搜索可查找发票、项目或供应商，也可使用 `Ctrl+K`；macOS 使用 `Command+K`。

常用视图包括“疑似重复”和“已确认”。点击一张发票可进入详情页查看原件、OCR 信息、字段和明细行。

### 5. 校对与确认

进入“待校对”，按以下队列处理：

- **待人工确认**：核对 OCR 字段和明细，保存后确认
- **疑似重复**：进入详情对比号码、日期、金额和原件；该记录不能直接确认
- **识别失败**：检查原件质量、OCR 配置和额度后重新识别

普通待确认记录可以单张确认或批量确认。包含重复风险、识别失败或状态不符合要求的记录不能确认。系统后端已提供重复判定能力，但当前前端尚未提供“确认重复/忽略”操作入口，需要由管理员通过 API 处理或等待前端入口补齐。

建议重点检查发票代码、号码、日期、购销双方、价税金额和明细合计。确认后，记录会进入可稳定统计和导出的状态。

### 6. 创建和下载导出

进入“导出记录”，选择项目、发票状态和格式后创建任务：

- **XLSX**：包含发票、明细行和 OCR 元数据工作表，适合财务核对
- **CSV**：导出发票主表，适合导入其他系统
- **JSON**：保留发票、明细和 OCR 元数据的结构化内容
- **项目文件包 ZIP**：打包指定项目的发票原件、普通文件和 `manifest.json`

导出由后台 worker 异步生成。任务完成后会出现下载按钮；下载文件保留 7 天，过期或生成失败后可使用“重新创建”。普通用户只能查看自己的导出任务，财务和管理员可查看全部任务。

## 状态与处理建议

| 页面状态   | 含义                       | 建议操作                             |
| ---------- | -------------------------- | ------------------------------------ |
| 待人工确认 | OCR 已返回，但需要人工核对 | 编辑字段和明细后确认                 |
| 疑似重复   | 系统发现可能相同的发票     | 对比号码、金额、日期和原件后处理     |
| 识别失败   | OCR 请求或结果处理失败     | 检查文件、凭据、额度和 worker 后重试 |
| 已确认     | 已完成人工确认             | 可用于统计和正式导出                 |
| 已归档     | 记录已移出日常处理范围     | 需要时从筛选结果中查看               |

## 运维与数据安全

### 停止和重启

停止服务但保留全部数据：

```bash
docker compose down
```

重新启动：

```bash
docker compose up -d
```

不要在日常停止、升级或故障排查时执行 `docker compose down -v`，`-v` 会删除 PostgreSQL、Redis、上传原件、临时文件和导出文件的数据卷。

### 数据位置与备份

Compose 使用以下命名卷：

| 数据卷          | 内容                                 | 是否必须备份     |
| --------------- | ------------------------------------ | ---------------- |
| `postgres_data` | 用户、项目、发票字段、配置和审计记录 | 是               |
| `app_uploads`   | 发票原件和项目文件                   | 是               |
| `app_exports`   | 已生成的导出文件                     | 按需             |
| `redis_data`    | 异步任务队列状态                     | 建议维护窗口保留 |
| `app_tmp`       | 临时处理文件                         | 否               |

生产环境应同时备份 PostgreSQL 和 `app_uploads`，并保存当前使用的 `POSTGRES_PASSWORD`、`APP_SECRET_KEY`、`OCR_CONFIG_ENCRYPTION_KEY`。只恢复数据库而不恢复上传卷，会导致原件无法预览或打包；更换 OCR 加密密钥后，已有 OCR 凭据将无法解密。

### 升级

升级前先备份数据库和上传文件，并记录当前密钥。更改docker-compose.yml里line 3的版本号，然后更新镜像版本并启动：

```bash
docker compose up -d
```

应用会自动执行缺失的数据库迁移，正常升级不会重建表或清空数据。不要在升级时重新生成数据库密码或 OCR 加密密钥。

### 公网访问

系统默认通过 HTTP 提供服务，适合内网或本机使用。需要公网访问时，请自行处理。

同时限制数据库和 Redis 不对公网开放。当前 Compose 仅发布应用的 Web 端口。

## 常见问题

### 上传后一直排队

先运行 `docker compose ps`，确认 `worker` 和 `redis` 正常，再查看：

```bash
docker compose logs --tail=200 worker redis
```

### OCR 提示失败或额度不足

由管理员在“设置”中检查启用的 OCR 配置、凭据、地域、QPS 和额度提醒。确认文件符合上传限制后，在“待校对”中重新识别。
腾讯云默认免费额度1000。

### 忘记管理员密码

如果仍有其他管理员，请在“用户管理”中重置密码。也可以在部署主机执行：

```bash
docker compose exec app invoice-app create-admin \
  --email admin@example.com \
  --password '至少12位的新密码' \
  --display-name '系统管理员'
```

该命令用于创建新的管理员；如果邮箱已存在且不是管理员，命令会拒绝覆盖。不要把真实密码留在 Shell 历史或运维记录中。

### 页面能打开，但就绪检查失败

`healthz` 只表示 Web 进程存在，`readyz` 失败通常表示 PostgreSQL 或 Redis 尚未就绪。使用以下命令查看依赖状态：

```bash
docker compose ps
docker compose logs --tail=200 postgres redis app
```

### 导出任务不生成文件

导出依赖 worker。确认 worker 正常运行且 `app_exports` 可写；项目 ZIP 还要求 `app_uploads` 中的原始文件完整存在。

## 本地构建与测试

本节面向参与开发和验收的人员，日常使用系统无需执行。

构建本地 `linux/amd64` 镜像：

```bash
docker buildx build --platform linux/amd64 --load \
  -t invoice-ocr-app:0.2.0 .
APP_IMAGE=invoice-ocr-app:0.2.0 docker compose up -d
```

运行后端测试：

```bash
cd backend
uv run --frozen --extra test pytest -v
```

运行前端测试和构建：

```bash
cd frontend
npm ci
npm test
npm run build
```

运行隔离的 Docker 端到端测试：

```bash
tests/e2e/run.sh
```

## 技术组成

- 前端：React 19、TypeScript、Vite
- API：FastAPI、SQLAlchemy、Alembic
- 异步任务：Celery、Redis
- 数据库：PostgreSQL 16
- 部署：Docker Compose，单一应用镜像同时提供 Web/API 和 worker
